# LLM Context & Routing Guide

This project uses a modular rules system. To provide accurate code, you MUST follow these steps:

## 0. GIT FLOW & CONVENTIONAL COMMITS (CRITICAL)

* **No Cyrillic:** Strictly prohibited anywhere in branch names and commit messages. English only.
* **Branch Naming:** Must follow the pattern: `feature/*`, `refactor/*`, `fix/*`, `chore/*`,
  `build/*`, `style/*`, `docs/*`, or `release/*`.
* **Commit Messages:** Strictly follow Conventional Commits format.
    * Allowed types: `feat`, `fix`, `chore`, `refactor`, `test`, `docs`, `style`, `ci`, `perf`,
      `build`, `revert`.
    * **CRITICAL:** The description part MUST start with a lowercase letter.
    * DO NOT do this: `feat(auth): Add login`
    * ALWAYS do this: `feat(auth): add login`

**STEP 1: ALWAYS READ GLOBAL RULES**
You must strictly adhere to the formatting, import ordering, and linting rules defined in:
`.rules/global.rules.md`

**STEP 2: READ DOMAIN-SPECIFIC RULES**
Depending on the task requested by the user, proactively use the `/read` command (or your
file-reading tool) to load the necessary architectural context:

* Modifying BLoCs, Cubits, or State? -> Read `.rules/state_management.rules.md`
* Working with Drift, DAOs, Migrations, or Models? -> Read `.rules/database.rules.md`
* Building UI, Widgets, or Theming? -> Read `.rules/theming.rules.md`
* App Architecture, Monorepo Packages, or Callbacks? -> Read `.rules/architecture.rules.md`
